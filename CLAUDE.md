# CLAUDE.md

이 저장소는 오케스트레이션 기반 AI 코딩 하네스(harness_v1)로 관리됩니다.

## 모드

- `/init` — 프로젝트 매니페스트(`.claude/project.yaml`)를 만들거나 갱신합니다. 새 프로젝트/레거시 여부는
  여기서 한 번 선언하고 이후 모든 세션이 그대로 참조합니다 (런타임에 추측하지 않습니다).
- `/create` — 생성 모드. 신규 프로젝트(`project.maturity: new`) 전용. 아키텍처 제안서를 쓰고,
  **사람이 직접 `runs/<run>/APPROVED` 파일을 만들어야만** 스캐폴딩을 진행합니다 — 이 승인 게이트는
  `approval_gate.py` 훅이 강제합니다(프롬프트 지시가 아닙니다).
- `/extend` — 기능추가 모드. 컨벤션 확인 → 영향범위 분석(대상이 있을 때) → 스펙에 대한 가벼운
  사람 확인(대화창에서 진행 승인, 훅으로 강제되지 않음) → 생성으로 갑니다.
- `/maintain` — 유지보수 모드. 재현 테스트로 버그를 먼저 고정 → 영향범위 분석(항상 필수) → 최소
  변경 스펙에 대한 가벼운 사람 확인 → 최소 변경으로 수정 → 재현 테스트·기존 테스트 통과 확인 →
  원인을 결정 로그에 기록.

## 원칙

1. **공유 코어 + 모드별 정책** — 도구·훅·서브에이전트는 공유하고, 프롬프트(커맨드)만 모드별로 다릅니다.
2. **규칙은 훅으로 강제한다** — 테스트/린트/브랜치 보호/아키텍처 승인은 프롬프트 지시가 아니라
   `.claude/hooks/`의 스크립트가 강제합니다.
3. **오케스트레이터는 지시만, 생성은 서브에이전트가** — 대량 생성은 `code-generator`(기본 Haiku)에게
   맡기고, 메인 세션(오케스트레이터)은 스펙 작성과 판단만 합니다.

## 서브에이전트

| 이름 | 기본 모델 | 역할 |
|---|---|---|
| `code-generator` | haiku | 명확한 스펙을 정확히 구현 |
| `impact-analyzer` | haiku | 심볼 참조·의존성 전수 검색 (판단 없이 목록만) |
| `verifier` | sonnet | 훅으로 판정 불가능한, 해석이 필요한 검증만 |

## 컨벤션

(프로젝트별 컨벤션은 `.claude/project.yaml`의 `conventions.notes`에 기록하고, `/init`이 여기 요약을
추가합니다.)

## 결정 로그

`.claude/decisions/`에 모드/아키텍처 관련 결정을 기록합니다. 새 결정이 생기면 파일을 추가하세요.

## 게시판

`runs/<run-id>/summary.md`가 진행 중인 작업의 현재 상태입니다. 형식은 `runs/README.md` 참고.
