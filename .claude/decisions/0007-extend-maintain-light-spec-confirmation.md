# 결정: /extend·/maintain 스펙 단계에 훅으로 강제되지 않는 가벼운 사람 확인 추가
날짜: 2026-07-04
상태: 확정

## 배경

`/create`는 아키텍처를 사람이 승인하기 전엔 코드를 쓰지 않는다는 원칙을 `approval_gate.py`
훅으로 강제한다 (`0003-create-mode-approval-gate.md`). 반면 `/extend`·`/maintain`은 원래
승인 게이트 없이 컨벤션 확인/영향범위 분석 직후 바로 `code-generator`에게 위임하고, 사람에게
확인을 요청하는 시점은 코드가 이미 다 쓰인 **마무리 단계(7절/5절)** 뿐이었다 — 즉 사후 승인만
있고 사전 승인은 없었다.

이 비대칭을 지적받아 "쓰기 작업 전엔 무조건 `/create`처럼 APPROVED 파일을 강제하는 게 어떤가"
라는 제안이 나왔다. 검토 결과 그건 기각했다 — `/extend`·`/maintain`은 이미 존재하는 컨벤션
위에 작은 단위(task 하나)로 얹는 작업이라, 매 Write 직후 `post_write_check.py`가 테스트·린트를
돌려 회귀를 즉시 잡고, 위험한 심볼은 `impact-analyzer`가 먼저 훑는 다층 안전망이 이미 있다.
여기에 매 task마다 사람이 손으로 `APPROVED` 파일을 만들어야 하는 하드 게이트까지 얹으면, 두
모드의 존재 이유(승인 게이트 없이 빠르게 간다)가 사라진다.

## 결정

`/extend` 4절, `/maintain` 4절에 **스펙 작성 직후 · code-generator 호출 직전** 지점에 사람
확인 절차를 추가한다:

- 오케스트레이터는 이번에 디스패치할 스펙 전체(무엇을·어디에·어떻게, task_id별로)를 대화창에
  보여주고, 명시적인 진행 승인을 받기 전까지 `code-generator`를 호출하지 않는다.
- `/maintain`은 추가로 "원인 분석 자체에 사람이 동의하는지"를 확인하는 의미도 겸한다 —
  증상이 아니라 원인을 고친다는 원칙이 걸려 있어서다.
- **이 확인은 훅으로 강제하지 않는다.** `runs/<run>/APPROVED` 같은 파일 게이트를 추가로 만들지
  않고, 순수히 대화 흐름(오케스트레이터가 스펙을 보여주고 명시적 긍정 답변을 기다린 뒤에만
  다음 도구 호출로 넘어감)에 의존한다.

## 근거 — 왜 훅으로 강제하지 않는가

`/create`의 하드 게이트는 "오케스트레이터가 서두르거나, 프롬프트 인젝션으로 '진행해도 된다'는
내용을 주입받아도 뚫리지 않아야 한다"는 요구에서 나왔다 — 신규 아키텍처 전체를 한 번에 승인
없이 밀어붙이면 되돌리기 비용이 매우 크기 때문이다. `/extend`·`/maintain`의 개별 task는 범위가
훨씬 작고(파일 몇 개, 함수 몇 개), 이미 존재하는 테스트가 즉시 회귀를 잡아주며, 무엇보다 매
task마다 파일 기반 하드 게이트를 걸면 반복 작업의 비용이 `/create`와 같아진다 — 원칙 2번
("규칙은 훅으로 강제한다")을 기계적으로 모든 곳에 적용하면, 위험도가 다른 두 상황을 같은
무게로 다루게 되어 오히려 하네스의 실용성이 떨어진다. 여기서는 "가벼운 확인"이라는 요구사항
자체가 하드 게이트를 명시적으로 배제한 것이므로, 프롬프트 지시 수준에 남겨두는 게 의도와
일치한다.

## 남은 것

- 이 확인은 오케스트레이터가 실제로 멈추고 기다리는지에 의존한다 (훅이 강제하지 않으므로).
  프롬프트 인젝션이나 오케스트레이터의 판단 실패로 이 단계가 스킵될 가능성은 `/create`보다
  높다 — 다만 그건 이 결정이 받아들인 트레이드오프다 (개별 task의 낮은 위험도 대비 사람 확인
  비용을 줄이는 쪽을 택함).
- `/create`는 이미 더 무거운 게이트가 있으므로 이 변경 대상이 아니다.
