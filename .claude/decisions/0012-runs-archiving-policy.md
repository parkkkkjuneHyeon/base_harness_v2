# 결정: `runs/` 아카이빙은 수동 컨벤션으로, 전용 도구는 미리 안 만든다
날짜: 2026-07-04
상태: 확정

## 배경

run 폴더의 아카이빙/정리 정책이 없었다 (`IMPROVEMENTS.md` 5번). 완료된 run이 계속 쌓이면
`runs/` 아래가 지저분해지고 `runs/.current`만 의미 있는 상태가 된다.

## 검토한 대안과 기각 이유

`/archive-runs` 같은 전용 커맨드나, run 생성 개수 기준 자동 압축 스크립트를 검토했지만
기각했다. 지금 시점엔 이게 실제로 아픈 문제라는 근거(예: run이 수십 개 쌓여서 탐색이
느려졌다는 보고)가 없다 — 가상의 미래 요구사항에 미리 도구를 만들지 않는다는 원칙
(`.claude/decisions/0011`과 같은 논리, `IMPROVEMENTS.md` 11번도 동일)을 여기도 적용했다.

## 결정

`runs/README.md`에 수동 컨벤션만 문서화했다: 완료된 run은 `runs/archive/`로 사람이 직접
옮기고, 필요하면 `agents/`의 상세 보고만 지워 용량을 줄인다. `summary.md`는 감사 기록이므로
남긴다.

## 근거

`summary.md`/`agents/*.md`는 사람이 감사 목적으로 버전 관리에 남기길 권장하는 파일이라
(`runs/README.md` 기존 문서), 자동 삭제/압축 도구가 실수로 감사 기록을 지우면 되돌리기
어렵다. 정책 자체가 아직 확정적이지 않은 상태(몇 개부터 "많다"고 볼지, 압축 기준이 뭔지)에서
도구부터 만들면 잘못된 정책을 자동화하게 될 위험도 있다.

## 남은 것

- 실제로 이 수동 작업이 반복되어 고통스러워지면, 그때 전용 커맨드나 스크립트를 검토한다.
