# 결정: /init 문서 보강 두 건 — 커밋 분리 안내(14a), pytest exit 5 회피(19b)
날짜: 2026-07-05
상태: 확정

## 1. `/init` 5단계 커밋 분리 안내 (`IMPROVEMENTS.md` 14번, 선택지 a 채택)

**배경**: `/init` 5단계의 `models.*` 오버라이드는 `.claude/agents/*.md`(pre-commit의 "하네스
인프라" 버킷)를 수정하는데, 같은 `/init` 흐름의 4단계가 `project.yaml`(그 외 버킷)도
수정한다. 둘을 한 커밋에 넣으면 `.githooks/pre-commit`이 차단한다 — `0008`이 agents 파일을
"프로젝트 내용이 절대 안 섞이는 순수 인프라"로 분류한 것과 실제 워크플로우의 모순.

**결정**: `init.md` 5단계에 "두 커밋으로 나눠 커밋하라"는 안내를 추가하는 것(선택지 a)으로
해결했다. 선택지 b(agents를 하네스 버킷에서 제외)는 기각 — model 프론트매터 한 줄 때문에
에이전트 role 정의 전체의 cherry-pick 보호를 포기하는 건 배보다 배꼽이 크다. agents 파일에서
프로젝트별 값은 `model:` 한 줄뿐이고, 그마저 대부분의 프로젝트는 기본값을 쓴다
(오버라이드는 선택 사항).

## 2. `commands.test`의 pytest exit 5 함정 안내 (`IMPROVEMENTS.md` 19번, 선택지 b 채택)

**배경**: pytest는 수집된 테스트가 0개면 exit 5를 반환하고, `post_write_check.py`는 non-zero를
전부 실패로 본다. 테스트가 아직 없는 레거시 프로젝트에서 `/init`이 `commands.test`를 채우는
순간 모든 소스 Write가 게이트 실패한다. 실사고: 이 세션에서 템플릿 브랜치의 README 작성 중
정확히 이 조합(exit 5)으로 게이트가 실패했다 (그 사건의 근본 원인이던 "소스 밖 파일에도
테스트 실행"은 13번에서 허용 목록 전환으로 별도 해결).

**결정**: `init.md` 3단계에 "테스트가 실제로 하나도 없으면 `commands.test`를 비워두고, 첫
테스트가 생긴 뒤에 채워라"는 안내를 추가하는 것(선택지 b)으로 해결했다. 선택지 a(훅에서
pytest exit 5를 특별 취급)는 기각 — 테스트 러너마다 exit code 의미가 달라서 명령 문자열에
"pytest"가 포함됐는지로 판단하는 휴리스틱이 되는데, 훅은 단순하게 유지한다는 원칙
(`0001`의 verifier 경계선과 같은 논리)에 어긋난다.
