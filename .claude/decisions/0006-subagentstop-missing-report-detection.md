# 결정: SubagentStop 페이로드로 보고 누락 자동 감지
날짜: 2026-07-04
상태: 확정

## 배경

`update_board.py`(`SubagentStop` 훅)는 원래 "보고를 빼먹었는지" 판정하지 못한다는 알려진
한계가 있었다 — 훅 입력엔 `agent_type`만 있고 지금 끝난 호출이 어떤 `task_id`에 해당하는지
알 방법이 없다고 가정했기 때문이다. 그래서 `extend.md`/`create.md`/`maintain.md`는 전부
"디스패치한 task_id 개수와 `summary.md` 행 수를 오케스트레이터가 직접 대조하라"는 수동 절차로
이 한계를 우회했다 (`0005-general-purpose-agent-fallback.md` 참고 — 별개 사안).

이 가정을 실제 Claude Code 환경에서 검증해보니 사실이 아니었다. `update_board.py`에 디버그
덤프를 잠깐 추가해 실제 `SubagentStop` 페이로드를 확인한 결과, 다음 필드가 이미 들어 있었다:

- `agent_id` — 이번에 끝난 호출의 고유 ID (병렬 호출도 서로 구분됨)
- `background_tasks[]` — 그 세션에서 추적 중인 서브에이전트 목록. 각 항목에 `id`(=agent_id와
  매칭), `agent_type`, 그리고 **`description`**(Agent 도구 호출 시 오케스트레이터가 직접 지정한
  라벨 문자열)이 들어 있음

즉 오케스트레이터가 이미 매번 채워 넣는 `description` 파라미터가 훅까지 그대로 전달된다.

## 결정

1. Agent 호출 시 `description`을 `"[<task_id>] <짧은 설명>"` 형식으로 쓰는 걸 규칙으로 삼는다
   (`extend.md`/`create.md`/`maintain.md`의 각 디스패치 절에 명시).
2. `update_board.py`가 `payload["agent_id"]`로 `background_tasks[]`에서 자기 항목을 찾고,
   `description`에서 정규식 `^\[([A-Za-z0-9_]+)\]`로 task_id를 파싱한다.
3. `agents/<agent_type>_<task_id>.md`가 실제로 존재하는지 확인한다. 없으면
   `runs/<run>/agents/.missing.json`에 기록해두고, `summary.md`에 `## ⚠ 보고 누락 감지` 섹션으로
   노출한다. 나중에 (다른 훅 호출 시점에) 해당 파일이 생기면 자동으로 목록에서 빠진다.
4. `description`이 `[task_id]` 규칙을 안 따르면(과거 호출, 규칙 미준수) 감지를 조용히
   건너뛴다 — 오탐보다 무동작이 낫다는 원칙. 이 경우 기존의 수동 대조 절차가 fallback으로
   남아 있다(문서에도 그렇게 명시).

**매니페스트 파일(`DISPATCHED`) 방식은 채택하지 않았다** — 오케스트레이터가 디스패치 시점에
별도 파일을 추가로 써야 해서 매니페스트와 실제 디스패치 사이에 드리프트가 생길 여지가 있는
반면, `description` 방식은 이미 존재하는 파라미터에 명명 규칙만 얹는 것이라 별도 상태 파일을
디스패치 시점에 만들 필요가 없다.

## 검증

임시 테스트 run(`hook-missing-detect-test`, 검증 후 삭제함)에서 두 가지를 확인했다:

1. `description: "[t1] ..."`로 호출했지만 서브에이전트가 스스로 다른 task_id
   (`dummy-hook-check`)로 보고서를 써버린 경우 — 예상대로 `verifier_t1`이 누락으로 잡혔다.
   (이건 원래 테스트하려던 "아예 안 씀" 케이스보다 더 실전적인 실패 모드였다 — verifier가
   시스템 프로토콜상 보고서 작성 자체는 거부하지 않고, 대신 task_id를 자기 마음대로 골랐다.)
2. 같은 task_id(`t1`)로 정확히 보고서를 작성하게 하자, 다음 훅 호출에서 `.missing.json`이
   빈 `{}`로 돌아오고 `summary.md`의 경고 섹션이 사라졌다.

## 남은 것

- 이 메커니즘은 `description` 규칙 준수에 의존한다. 오케스트레이터(프롬프트 지시)가 이 규칙을
  까먹으면 감지가 조용히 꺼진다 — 훅으로 강제되는 게 아니라 여전히 프롬프트 지시 수준이다.
  다만 무동작으로 안전하게 실패하므로(오탐 없음), 과거의 수동 대조 절차를 대체하되 없애지는
  않았다.
- 서브에이전트가 오케스트레이터가 지정한 task_id를 무시하고 자기 마음대로 짓는 경우(검증에서
  실제로 관찰됨)는 이 메커니즘이 정확히 "누락"으로 잡아준다 — 사실 이것도 이 메커니즘이 잡아야
  할 실패 모드 중 하나였으므로 의도대로 동작한 것으로 본다.
