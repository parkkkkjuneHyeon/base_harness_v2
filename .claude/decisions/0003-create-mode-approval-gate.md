# 결정: /create 구현 + 아키텍처 승인 게이트를 훅으로 강제
날짜: 2026-07-03
상태: 확정

## 배경

구현 리뷰 중 4가지 gap을 발견했다. 그중 "생성 모드 전용 커맨드(`/create`)가 없다"와 "사람 승인
게이트가 프롬프트 지시일 뿐 훅으로 강제되지 않는다"는 서로 얽혀 있었다 — `/create`의 핵심
차별점 자체가 "아키텍처를 사람이 승인하기 전엔 코드를 안 쓴다"이므로, `/create`를 제대로
만들려면 그 승인을 실제로 강제하는 방법을 같이 설계해야 했다.

## 결정

`/create`를 구현하면서, 승인 게이트를 `.claude/hooks/approval_gate.py`(PreToolUse)로 강제한다.

1. `runs/<run>/APPROVED` 파일은 Write/Edit 어떤 경우에도 하네스 툴로 만들 수 없다 — 항상 차단.
   승인은 사람이 자기 터미널/에디터로 직접 그 파일을 만들어야만 성립한다. 오케스트레이터가
   대화창의 "승인했다"는 말을 듣고 자기가 그 파일을 대신 써버리면 승인이 무의미해지므로, 그
   경로 자체를 원천 차단했다.
2. `runs/<run>/MODE`가 `create`인 run에서만, `project.source_root` 아래 Write/Edit을
   `APPROVED` 파일 존재 여부로 게이팅한다. `/extend`는 `MODE`를 `extend`로 써서 이 게이트를
   받지 않는다.

## 근거 — 왜 파일 기반 게이트인가

PreToolUse 훅은 도구 호출만 보고, "사람이 채팅에 뭐라고 썼는지"는 알 수 없다. 따라서 승인
여부를 판단할 수 있는 유일한 방법은 "Claude Code 도구 호출 경로 밖에서만 만들어질 수 있는
파일의 존재"뿐이다. 오케스트레이터(LLM) 스스로 그 파일을 만들 수 있게 두면, 프롬프트 지시가
전부 무너졌을 때(프롬프트 인젝션, 판단 착오 등) 게이트도 같이 무너진다 — 그래서 규칙 1로
"그 파일은 어떤 상황에서도 하네스 도구로 못 만든다"를 무조건 걸었다. 이러면 오케스트레이터가
아무리 서두르거나 잘못 판단해도, 사람이 실제로 파일을 만들지 않는 한 `project.source_root`
아래에는 아무것도 못 쓴다.

## 남은 한계

- 사람이 승인 파일을 만드는 행위 자체는 검증하지 않는다 (예: 내용이 비어있어도 존재하면 통과).
  악의적인 상황을 막기 위한 장치가 아니라, 정상적인 협업에서 "승인 없이 그냥 진행"을 막기
  위한 장치다.
- 나머지 3개 gap(테스트/CI 골격 자동생성, `extend.md` 2절이 신규 프로젝트를 고려 안 함)은
  이번 라운드에서 다루지 않았다 — 우선순위 낮게 그대로 둔다.

## 2026-07-04 추가: 삭제 후 복원

이 훅(`approval_gate.py`)과 이 결정 로그는 한 세션에서 "하네스 보완작업" 정리 과정 중 실수로
같이 삭제됐다가(`settings.json`의 `PreToolUse` 연결도 함께 빠짐), 다른 세션에서 `/create`
문서(`create.md`, `CLAUDE.md`)가 여전히 이 훅을 전제로 설명하고 있는 걸 발견하고 원래대로
복원했다. `CLAUDE.md` 원칙 2번도 같이 확인할 것 — "아키텍처 승인"이 빠진 채로 남아있으면 이
결정과 다시 어긋난다.
