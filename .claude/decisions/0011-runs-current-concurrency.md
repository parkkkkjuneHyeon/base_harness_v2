# 결정: `runs/.current` 동시성 문제는 세션별 포인터 대신 가벼운 사전 확인으로
날짜: 2026-07-04
상태: 확정

## 배경

`runs/.current`는 전역 단일 포인터라, 같은 저장소를 두 세션(사람 둘 또는 같은 사람의 세션
둘)이 동시에 열면 나중에 만든 run이 먼저 것의 포인터를 덮어쓴다. 그러면 `update_board.py`/
`approval_gate.py`가 엉뚱한 run을 기준으로 동작한다 (`IMPROVEMENTS.md` 6번).

## 검증

이번 세션에서 실측한 `PreToolUse`/`SubagentStop` 훅 페이로드에 `session_id`가 이미 들어있는
걸 확인했다 (`0006`, `0009` 결정 로그의 실측 결과와 동일 페이로드). 따라서
`runs/.current-<session_id>` 같은 세션별 포인터 분리는 기술적으로 가능하다.

## 결정

세션별 포인터 파일을 새로 만들지 않는다. 대신 `/extend`·`/create`·`/maintain`의 "1. run 폴더
생성" 단계에 가벼운 사전 확인을 추가했다: 새 run을 만들기 전에 `runs/.current`가 가리키는
기존 run의 `summary.md`에 `done`이 아닌 status가 남아있으면, 사람에게 "이어서 할까요, 새로
시작할까요?"라고 먼저 물어본다.

## 근거

세션별 포인터 파일은 실제로 만들 수 있지만 새로운 문제를 만든다 — 세션이 비정상 종료되면
포인터 파일이 고아로 남고, "지금 활성 run이 뭔지" 확인할 때 사람이 어느 파일을 봐야 하는지
더 헷갈려진다. 반면 이 저장소를 동시에 여는 다중 세션 자체가 아직 실제로 문제를 일으킨
사례가 없는 가상의 시나리오다. `.claude/decisions/0008`(하네스/프로젝트 커밋 분리에서
"decisions/ 전부는 안 막는다")과 `IMPROVEMENTS.md` 11번("반복되면 그때 만든다")과 같은
논리 — 확실하지 않은 미래 요구사항에 미리 인프라를 만들지 않는다. 가벼운 사전 확인은 새
run 생성 시점에 사람이 한 번 더 보게 만드는 것만으로 실질적인 사고(엉뚱한 run에 덮어쓰기)를
대부분 막는다.

## 남은 것

- 이 확인은 오케스트레이터가 매턴 `summary.md`를 읽고 판단하는 프롬프트 지시이지, 훅으로
  강제되지 않는다. 세 커맨드 문서 다 놓치지 않고 반영했는지 확인할 것.
- 실제로 다중 세션 충돌이 반복해서 발생하면, 그때 `session_id` 기반 분리를 다시 검토한다.
