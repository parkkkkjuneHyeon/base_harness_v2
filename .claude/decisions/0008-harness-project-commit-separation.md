# 결정: 하네스와 프로젝트를 같은 저장소에 두되, 커밋 단위로 분리 강제
날짜: 2026-07-04
상태: 확정

## 배경

이 저장소는 `main`(재사용 가능한 하네스 템플릿)과 `project/<이름>`(실제 프로젝트) 브랜치를
같은 git 저장소 안에서 운영한다. 팀원이 `project/<이름>` 브랜치에서 작업하다가 하네스도 같이
고치는 일이 자주 있다(훅 버그 수정, 커맨드 문구 개선 등) — 실제로 이번 세션에서도 그런
일이 반복적으로 일어났다. 문제는 그 개선이 프로젝트 코드와 같은 커밋에 섞여 있으면, 나중에
그 개선만 `main`이나 다른 프로젝트 브랜치로 옮기고 싶어도 `git cherry-pick`이 프로젝트 전용
내용까지 끌고 가서 깨끗하게 옮길 수 없다는 것이다. 이번 세션에서 `main`과 `project/book-graph-rag`
사이의 커맨드 문서가 서로 다른 시점의 개선을 각각 담은 채 갈라져서, 수동으로 diff를 떠가며
병합해야 했던 것도 근본 원인이 같다 — 하네스 변경과 프로젝트 변경이 커밋 단위로 분리되어
있지 않았다.

## 검토한 대안과 기각 이유

**git submodule/subtree로 하네스를 완전히 별도 저장소로 분리**하는 안을 검토했지만 기각했다.
지금은 "프로젝트 = 브랜치"라는 가벼운 운영 모델을 쓰고 있는데, submodule/subtree는 이 모델
자체를 별도 저장소 + 포인터 관리 방식으로 바꿔야 해서 팀원에게 학습 비용이 크다. 커밋
히스토리가 완전히 분리된다는 장점은 있지만, 지금 필요한 건 "같은 저장소 안에서 하네스 개선을
깨끗하게 골라낼 수 있는 것"이지 완전한 물리적 분리가 아니다.

## 결정

같은 저장소 구조는 그대로 두고, `.githooks/pre-commit`(git pre-commit 훅)으로 다음을 강제한다:

- 순수 하네스 인프라 파일(`.claude/agents/`, `.claude/commands/`, `.claude/hooks/`,
  `.claude/settings.json`, `.githooks/`, `.gitignore`, 루트 `README.md`, `runs/README.md`)과
  그 외 파일이 **한 커밋에 같이 스테이징되어 있으면 커밋을 막는다.**
- 이 훅은 `.git/hooks/`가 아니라 저장소 안의 `.githooks/`에 있어서 클론만으로는 활성화되지
  않는다. 팀원이 `git config core.hooksPath .githooks`를 최초 1회 실행해야 한다.

**의도적으로 강제 대상에서 뺀 것**: `CLAUDE.md`, `.claude/decisions/`, `.claude/project.yaml`.
- `CLAUDE.md`는 컨벤션 섹션에 `/init`이 프로젝트별 컨벤션을 적어 넣는다 — 파일 전체를
  "하네스"로 분류하면, 정상적인 `/init` 실행 결과물도 분리 대상이 되어버린다.
- `.claude/decisions/`는 `/maintain.md` 7절이 프로젝트 버그의 원인·근거를 여기 적으라고
  명시한다 — 코드 수정과 같은 커밋에 있는 게 오히려 정상 워크플로우다.
- `.claude/project.yaml`은 필드 구조는 하네스 것이지만 값 자체가 항상 프로젝트 전용이다.

이 셋을 강제 목록에 넣으면, `/init`·`/maintain`의 정상적인 한 걸음(코드 수정 + 결정 기록을
같이 커밋)마다 훅이 걸려서 실용성이 떨어진다. 대신 이 셋을 하네스 개선 목적으로 다른
브랜치에 옮기고 싶을 땐 diff를 사람이 직접 보고 프로젝트 전용 내용을 걷어내야 한다 — 완벽한
자동 분리보다, 오탐 없이 확실한 부분만 기계적으로 강제하는 쪽을 택했다.

## 근거

이 저장소의 기존 철학("기계적으로 판정 가능한 것만 훅으로 강제하고, 나머지는 사람 판단에
맡긴다" — `.claude/decisions/0001-open-questions-round1.md`의 Verifier 경계선 결정과 동일한
논리)을 커밋 단위 분리에도 그대로 적용했다. 순수 인프라 파일은 "프로젝트 코드와 섞일 이유가
없다"는 게 항상 참이므로 기계적으로 막아도 오탐이 없지만, `CLAUDE.md`/`decisions/`/
`project.yaml`은 "섞이는 게 정상인 경우"가 실제로 있어서 기계적 판정이 위험하다.

## 남은 한계

- `.githooks/pre-commit`은 git 표준 훅 디렉토리가 아니라서 `core.hooksPath` 설정을 깜빡하면
  전혀 작동하지 않는다 — 이것도 프롬프트 지시가 아니라 사람이 최초 1회 해야 하는 설정이라는
  점에서 완전한 강제는 아니다. `/init` 절차에 이 설정 확인을 추가하는 것도 고려할 수 있지만
  이번 라운드에서는 다루지 않았다.
- `CLAUDE.md`/`decisions/`/`project.yaml`을 다른 브랜치로 옮길 때의 "사람이 diff 보고 걷어내기"는
  여전히 수동 작업이다 — 도구로 자동화하지 않았다.
