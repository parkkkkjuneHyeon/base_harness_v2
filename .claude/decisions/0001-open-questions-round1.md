# 결정: 1차 설계의 열린 질문 4가지 해소
날짜: 2026-07-03
상태: 확정

## 1. Verifier가 모델을 써야 하는 경계선

**결정**: 기계적으로(스크립트로 if-else 표현 가능) 판정되는 것은 전부 훅. verifier(모델)는
다음 세 가지처럼 해석이 필요한 경우에만 호출한다.

1. API 하위 호환성 — 사용 패턴 이해가 필요
2. 실패한 테스트가 이번 변경 때문인지 원래 flaky했는지
3. 문서화되지 않은 기존 설계 의도와의 정합성

`.claude/agents/verifier.md`의 description과 본문에 이 경계를 명시해 프롬프트 차원에서도
강제한다.

## 2. 브랜치 자동 생성 시 네이밍 규칙

**결정**: 자동화하지 않는다. 하네스는 "보호된 브랜치에서 쓰기 작업 중"이라는 사실만
`branch_guard.py`(PreToolUse 훅)로 차단하고, 브랜치를 만들고 이름 짓는 것은 사람이 직접 한다.

## 3. 생성 모드 아키텍처 게이트 UX

**결정**: 보류. `/extend`(기능추가)를 먼저 완성하고, 생성 모드는 우선순위 3위 그대로 다음
단계에서 설계한다.

## 4. 서브에이전트 모델 배정 오버라이드 허용 여부

**결정**: 허용한다. `.claude/project.yaml`의 `models.*`에 값을 채우면 `/init`이 해당
서브에이전트 파일의 `model:` 프론트매터를 그 값으로 덮어쓴다. 비워두면 각 서브에이전트 파일의
기본값(code-generator/impact-analyzer는 haiku, verifier는 sonnet)을 그대로 쓴다.

## 근거

팀마다 비용 민감도가 다르다는 문제를, 공유 코어(서브에이전트 파일 자체)는 건드리지 않고
매니페스트라는 정책 레벨의 설정값으로 해결했다 — "공유 코어 + 모드별/프로젝트별 정책"이라는
1차 설계 원칙과 일치한다.
