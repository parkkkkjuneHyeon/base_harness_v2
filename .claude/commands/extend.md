당신은 이 작업의 **오케스트레이터**입니다. 코드를 직접 작성하지 않습니다 — 생성은
`code-generator` 서브에이전트에게, 참조 검색은 `impact-analyzer` 서브에이전트에게 맡기고,
당신은 스펙을 쓰고 결과를 검토·통합하고 판단합니다.

**`<run>` 표기 규칙**: 이 문서에서 `<run>`은 1단계에서 만든 실제 run 폴더 이름을 가리키는
자리표시자입니다 (`runs/README.md` 참고). 서브에이전트에게 보고 경로를 지시할 때는 이 값을
실제 이름으로 치환해서 넘깁니다 — 확실하지 않으면 `runs/.current`를 다시 읽습니다.

**서브에이전트 호출 규칙**: 아래에서 `code-generator`/`impact-analyzer`를 Agent 도구로 부를
때마다, `description`을 항상 `"[<task_id>] <짧은 설명>"` 형식으로 씁니다 (예: `"[t2] neo4j_uri
잔여 참조 검색"`). `update_board.py` 훅이 이 접두사로 보고 누락을 자동 감지합니다 — 5절 참고.

작업 대상: $ARGUMENTS

## 0. 사전 확인

- `.claude/project.yaml`이 없거나 `project.maturity`가 비어 있으면 먼저 `/init`을 실행하라고
  안내하고 멈춥니다.
- 지금 git 브랜치가 `.claude/project.yaml`의 `branches.protected`에 있으면, 사람에게 기능 전용
  브랜치를 만들고 전환해달라고 요청하고 멈춥니다. (`branch_guard.py` 훅이 실제 쓰기 시점에도 한
  번 더 막습니다.)

## 1. run 폴더 생성

**타임스탬프는 직접 만들어내지 않습니다** — 반드시 `date` 명령으로 실제 값을 받아옵니다. Bash로
아래 명령을 실행합니다 (`SLUG`만 이번 작업에 맞는 짧은 이름 — 소문자·숫자·하이픈만 — 으로
바꿉니다):

```bash
SLUG="기능-슬러그"
RUN_ID="$(date +%Y%m%d-%H%M%S)_${SLUG}_$(printf '%04x' $((RANDOM % 65536)))"
mkdir -p "runs/$RUN_ID/agents"
printf '%s' "$RUN_ID" > runs/.current
printf 'extend' > "runs/$RUN_ID/MODE"
echo "RUN_ID=$RUN_ID"
```

이 명령의 출력(`RUN_ID=...`)이 이후 이 문서에서 말하는 `<run>`의 실제 값입니다. (`MODE`가
`create`가 아니므로 `approval_gate.py`의 승인 게이트는 이 run에 적용되지 않습니다 — `/extend`는
원래 승인 절차 없이 진행합니다.) 이후 모든 서브에이전트 보고는 이 폴더 밑에 쌓이고,
`update_board.py` 훅이 `summary.md`를 자동으로 갱신합니다.

## 2. 컨벤션·통합 지점 파악

- `CLAUDE.md`와 `.claude/project.yaml`의 `conventions.notes`를 읽습니다.
- 비슷한 기존 기능 1~2개를 직접 읽어 네이밍·폴더 구조·에러 처리·테스트 스타일 패턴을 파악합니다.
- 이 기능이 기존 코드의 어디에 연결되는지(통합 지점)를 정합니다.

## 3. 영향범위 분석 (추가·변경·삭제 대상이 있을 때)

바꾸거나 지우려는 심볼이 있으면, 코드를 쓰기 전에 `impact-analyzer`에게 위임합니다:

- `task_id`를 하나 정합니다 (예: `t1`).
- Task 프롬프트에 검색 대상 심볼과, 정확한 보고 경로(`runs/<run>/agents/impact-analyzer_t1.md`)를
  명시합니다.
- 결과 보고서를 읽고, 참조가 남아있는데 삭제하려는 경우라면 **삭제를 취소하거나 deprecation
  경고를 먼저 붙이는 쪽으로 스펙을 조정**합니다.
- 코드베이스 검색만으로는 부족합니다 — 삭제 대상이 실제 운영에서 쓰이고 있는지는 하네스가 볼 수
  없는 신호이므로, 삭제가 포함된 작업은 마무리 단계(7절)에서 반드시 사람에게 확인을 요청합니다.

## 4. 스펙 작성 → 가벼운 확인 → Generator 위임

아래 항목을 전부 채운 스펙을 작성합니다 (이번 디스패치 라운드에서 독립적인 부분이 여러 개면
한 번에 다 씁니다):

- 입출력 계약 (함수/API 시그니처, 반환 타입, 에러 케이스)
- 지켜야 할 기존 컨벤션 (2절에서 파악한 것)
- 영향범위 분석 결과 (3절에서 나온 것, 있다면)
- 참조해야 할 기존 파일 목록
- **새로 만들 파일의 정확한 경로** — `project.source_root`(기본 `src`) 아래로, 오케스트레이터가
  직접 정해서 명시합니다. 서브에이전트는 이 경로를 스스로 정하지 않습니다.
- 정확한 보고 경로 (`runs/<run>/agents/code-generator_<task_id>.md`) — **위임마다 이번 run에서
  처음 쓰는 `task_id`를 새로 정합니다.** 3절에서 쓴 값과도 겹치면 안 됩니다 (예: 3절이 `t1`을
  썼다면 여기는 `t2`부터).

**code-generator를 호출하기 전에, 작성한 스펙 전체(무엇을·어디에·어떻게 만들지, task_id별로)를
사람에게 보여주고 진행 승인을 기다립니다.** 이건 `/create`의 `APPROVED` 파일 게이트와 달리
훅으로 강제되지 않습니다 — 기존 컨벤션과 테스트라는 안전망이 이미 있는 작은 단위 변경이라,
대화창에서 "진행하세요"라는 명시적 답을 받는 정도의 가벼운 확인으로 충분하다고 판단합니다.
사람이 스펙 수정을 요청하면 고쳐서 다시 보여주고, 명시적인 긍정 답변을 받기 전까지는
code-generator를 호출하지 않습니다.

승인을 받으면 하나 이상의 `code-generator` 서브에이전트에게 위임합니다 (독립적인 부분은
병렬로 dispatch해도 됩니다).

스펙이 부실하면 작은 모델이 잘못 채우고 재작업 왕복이 늘어난다는 점을 기억하세요 — 모호한 채로
넘기지 말고 여기서 명확히 합니다.

## 5. 결과 확인

- 각 Write/Edit 직후 훅이 자동으로 테스트·린트를 돌립니다 (실패하면 stderr로 알려줍니다) —
  당신이 따로 테스트 명령을 실행할 필요는 없습니다.
- 서브에이전트가 끝날 때마다 `runs/<run>/summary.md`가 자동 갱신됩니다.
- **`description`을 `"[<task_id>]"` 규칙대로 썼다면, 보고 누락은 훅이 자동으로 감지합니다.**
  `update_board.py`가 SubagentStop 페이로드에서 방금 끝난 호출의 task_id를 알아내 예상 보고
  파일이 실제로 있는지 확인하고, 없으면 `summary.md`에 `## ⚠ 보고 누락 감지` 섹션으로 표시해
  둡니다(보고가 들어오면 자동으로 사라집니다). 이 섹션이 보이면 해당 서브에이전트를 다시
  확인하세요. (`description` 규칙을 안 지켰거나 과거 방식으로 호출했다면 이 감지가 조용히
  건너뛰어지므로, 그럴 땐 디스패치한 task_id 개수와 `summary.md` 행 수를 직접 대조하세요.)
- `status: blocked`나 `failed`가 있으면 원인을 보고 재지시하거나, 하네스가 판단할 수 없는
  문제면 사람에게 에스컬레이션합니다.

## 6. 판단이 필요한 검증만 verifier에게

다음에 해당할 때만 `verifier`를 호출합니다 (그 외는 이미 훅이 처리했으므로 호출하지 않습니다):

- API 하위 호환성 판단이 필요할 때
- 실패한 테스트가 이번 변경 때문인지 원래 불안정했는지 구분해야 할 때
- 문서화되지 않은 기존 설계 의도와의 정합성 판단이 필요할 때

## 7. 마무리

- `runs/<run>/summary.md`가 전부 `done`인지 확인합니다.
- 삭제가 포함된 작업이었다면, 3절에서 미룬 "실제 운영에서 쓰이는지" 확인을 사람에게 요청합니다.
- 변경 요약과 함께 사람에게 최종 확인을 요청합니다 — 훅은 테스트/린트까지만 자동으로 걸러주고,
  최종 승인은 사람의 몫입니다.
