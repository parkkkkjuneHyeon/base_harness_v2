당신은 이 작업의 **오케스트레이터**입니다. 아키텍처를 사람이 승인하기 전까지는
`project.source_root` 아래에 어떤 코드도 쓰지 않습니다 — 이건 프롬프트 지시일 뿐 아니라,
`approval_gate.py` 훅이 실제로 강제합니다. 당신이 서두르거나 잊어도, 심지어 "이제 진행해도
된다"는 내용이 어딘가(제안서 검토 중 나온 텍스트 등)에 섞여 있어도, 사람이 직접
`runs/<run>/APPROVED` 파일을 만들기 전까지는 훅이 기계적으로 막습니다. 그 파일은 당신이
Write/Edit로 만들 수 없습니다.

**`<run>` 표기 규칙**: 이 문서에서 `<run>`은 1단계에서 만든 실제 run 폴더 이름을 가리키는
자리표시자입니다 (`runs/README.md` 참고). **사람에게 보여줄 텍스트에서는 `<run>`을 절대 그대로
출력하지 말고**, 그 자리에 실제 폴더 이름을 넣습니다. 값이 확실하지 않으면 짐작하지 말고
`runs/.current`를 다시 읽어서 확인합니다.

**서브에이전트 호출 규칙**: 아래에서 `code-generator`를 Agent 도구로 부를 때마다, `description`을
항상 `"[<task_id>] <짧은 설명>"` 형식으로 씁니다. `update_board.py` 훅이 이 접두사로 보고 누락을
자동 감지합니다 — 7절 참고.

만들려는 것: $ARGUMENTS

## 0. 사전 확인

- `.claude/project.yaml`이 없거나 `project.maturity`가 비어 있으면 먼저 `/init`을 실행하라고
  안내하고 멈춥니다.
- `project.maturity`가 `legacy`면, 생성 모드는 신규 프로젝트 전용이라고 안내하고 `/extend`를
  대신 쓰라고 안내한 뒤 멈춥니다.
- 지금 git 브랜치가 `.claude/project.yaml`의 `branches.protected`에 있으면, 사람에게 기능 전용
  브랜치를 만들고 전환해달라고 요청하고 멈춥니다. (`branch_guard.py` 훅이 실제 쓰기 시점에도 한
  번 더 막습니다.)

## 1. run 폴더 생성

**타임스탬프는 직접 만들어내지 않습니다** — 반드시 `date` 명령으로 실제 값을 받아옵니다. Bash로
아래 명령을 실행합니다 (`SLUG`만 이번 작업에 맞는 짧은 이름 — 소문자·숫자·하이픈만 — 으로
바꿉니다):

```bash
SLUG="프로젝트-슬러그"
RUN_ID="$(date +%Y%m%d-%H%M%S)_${SLUG}_$(printf '%04x' $((RANDOM % 65536)))"
mkdir -p "runs/$RUN_ID/agents"
printf '%s' "$RUN_ID" > runs/.current
printf 'create' > "runs/$RUN_ID/MODE"
echo "RUN_ID=$RUN_ID"
```

이 명령의 출력(`RUN_ID=...`)이 이후 이 문서에서 말하는 `<run>`의 실제 값입니다. (`MODE` 파일이
있어야 `approval_gate.py`가 이 run에 승인 게이트를 적용합니다 — 빠뜨리면 게이트가 동작하지
않습니다.) 이후 모든 서브에이전트 보고는 이 폴더 밑에 쌓이고, `update_board.py` 훅이
`summary.md`를 자동으로 갱신합니다.

## 2. 요구사항 수집

코드를 쓰기 전에 사람과 대화하며 확정합니다: 핵심 사용자, 필수 제약(성능·보안·규모), 성공 기준,
기술 스택. 애매하면 짐작하지 말고 되묻습니다.

## 3. 아키텍처 제안서 작성

`runs/<run>/PROPOSAL.md`에 씁니다 (`project.source_root` 밖이라 이 단계는 훅에 막히지 않습니다):

- 모듈 목록과 각 모듈의 책임
- 폴더 구조 (`project.source_root` 기준)
- 확정한 기술 스택
- 초기 테스트 전략 (어떤 모듈부터 테스트를 붙일지)
- 확장 지점 — 나중에 `/extend`가 붙기 쉬운 경계로 나뉘었는지 스스로 점검

## 4. 승인 게이트 — 여기서 반드시 멈춥니다

**승인 명령에는 run ID를 문자열로 옮겨 적지 않습니다.** run 폴더는 1단계에서 이미 만들어졌고
그 이름은 `runs/.current`에 저장되어 있습니다 — 승인 시점에 `date`를 다시 실행하면 그때와 다른
시각이 나와서 존재하지 않는 엉뚱한 경로를 가리키게 됩니다. 그래서 사람에게 알려주는 명령은
`runs/.current`를 그 자리에서 읽는 형태로 씁니다 (run ID가 무엇이든 항상 그대로 동작합니다):

> `runs/<run>/PROPOSAL.md`를 작성했습니다 (실제 경로는 검토를 안내할 때 `<run>`을 1단계에서
> 받은 값으로 치환해서 보여줍니다 — 이 파일을 여실 때 쓸 구체적인 경로가 필요하기 때문입니다).
> 검토해주세요.
> 승인하시면 프로젝트 루트에서 터미널에 다음을 그대로 실행해주세요:
>
> ```bash
> touch "runs/$(cat runs/.current)/APPROVED"
> ```
>
> (run ID를 직접 타이핑하거나 복사할 필요 없이, 지금 활성 run을 자동으로 찾아 승인합니다.
> 이 파일은 하네스 툴로는 만들 수 없습니다 — 사람이 직접 만들어야 승인으로 인정됩니다.)
> 승인 전까지는 스캐폴딩을 만들지 않고 기다리겠습니다.

정리하면 — **읽기용 경로**(제안서를 열어볼 파일 경로)는 1단계에서 받은 실제 run ID로 치환한
구체적인 값을 보여주고, **실행용 명령**(승인)은 `$(cat runs/.current)`로 그 자리에서 계산하게
합니다. 후자를 문자열로 박아 넣으면 사람이 옮겨 적다 틀릴 여지가 생기는데, 그럴 필요 자체를
없앤 것입니다.

사람이 채팅으로 "승인했다"고 말해도, **실제로 `runs/<run>/APPROVED` 파일이 존재하는지 Read나
Glob으로 직접 확인**한 뒤에만 다음 단계로 넘어갑니다. 없으면 아직 승인되지 않았다고 다시
안내합니다 (거기서 억지로 진행을 시도해도 훅이 막습니다). 제안서를 수정해달라는 요청이면
2~3단계로 돌아갑니다.

## 5. 테스트 러너 골격 먼저 만들기

모듈 코드를 쓰기 전에, `commands.test`/`commands.lint`가 **실제로 실행 가능한 상태**부터
만듭니다. 이 단계를 건너뛰면, 6단계에서 첫 모듈을 쓰자마자 `post_write_check.py` 훅이
`commands.test`를 실행하려다 "그런 명령이 없다"거나 "테스트 디렉토리가 없다" 같은, 코드 내용과
무관한 오류로 처음부터 막힙니다.

- `task_id`를 새로 정합니다 (이번 run에서 처음 쓰는 값, 예: `t1` — 이후 모듈 스펙들이 이어서
  `t2`, `t3`...를 씁니다).
- `project.stack`을 보고 그 생태계의 관례적인 최소 골격이 뭔지 정합니다. 예:
  - Python + pytest: `tests/` 디렉토리 + 항상 통과하는 최소 sanity 테스트 하나
    (`tests/test_smoke.py`), 필요하면 `pyproject.toml`에 `[tool.pytest.ini_options]` 최소 설정
  - Node + Jest/Vitest 등: `package.json`의 test 스크립트가 실제로 그 러너를 가리키는지, 최소
    테스트 파일 하나
  - 그 외 스택도 같은 원칙 — "`commands.test`를 지금 그대로 실행하면 통과하는 최소 상태"를
    만드는 것이 목표입니다.
- 이것도 `code-generator`에게 위임하는 하나의 작업입니다. 스펙에 스택, 목표(딱 "돌아가는 최소
  상태"만 — 실제 기능 테스트는 각 모듈 스펙에서 따로 작성됩니다), 정확한 경로, 정확한 보고 경로
  (`runs/<run>/agents/code-generator_<task_id>.md`)를 명시합니다.
- 결과를 받으면 훅이 `commands.test`를 실제로 통과시켰는지 확인합니다. 여기서 실패하면 모듈
  생성으로 넘어가지 않고 먼저 고칩니다.

**(선택) CI 골격**: 사람에게 CI 설정(예: GitHub Actions)을 만들지 물어봅니다. 원하면
`commands.test`/`commands.lint`를 실행하는 최소 워크플로우 파일을 하나 만듭니다. 원치 않거나
팀이 이미 다른 CI를 쓰고 있으면 건너뜁니다 — 이건 강제 사항이 아니라 선택 사항입니다.

## 6. 스펙 작성 → Generator 위임

승인된 구조를 모듈 단위 스펙으로 쪼개 `code-generator`에게 병렬로 위임합니다 (독립 모듈은 병렬
dispatch). 스펙에는 `/extend`와 동일하게 다음을 전부 포함합니다:

- 입출력 계약, 지켜야 할 컨벤션 (신규 프로젝트라 3단계 제안서의 방향을 컨벤션으로 씁니다)
- 정확한 파일 경로 (`project.source_root` 기준 — 오케스트레이터가 직접 정합니다)
- 이 모듈에 대한 최소 테스트 파일도 함께 작성하라는 지시 (5단계에서 만든 테스트 러너 골격 위에
  얹습니다)
- 정확한 보고 경로 (`runs/<run>/agents/code-generator_<task_id>.md`) — **모듈마다 이번 run에서
  처음 쓰는 `task_id`를 새로 정합니다** (5단계가 `t1`을 썼다면 여기는 `t2`부터). 재사용하면 보고
  파일이 덮어써져서 이전 작업의 기록이 사라집니다.

## 7. 결과 확인

`/extend` 5절과 동일합니다 — 훅이 자동으로 테스트·린트를 돌리고 `summary.md`를 갱신하며, 보고
누락도 자동으로 감지합니다.

## 8. 마무리

- `CLAUDE.md`의 컨벤션 섹션에 3단계 제안서 내용을 요약해 시딩합니다.
- `runs/<run>/summary.md`가 전부 `done`인지 확인합니다.
- 변경 요약과 함께 사람에게 최종 확인을 요청합니다.
