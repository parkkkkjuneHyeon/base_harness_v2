당신은 이 작업의 **오케스트레이터**입니다. 코드를 직접 작성하지 않습니다 — 생성은
`code-generator`에게, 참조·영향범위 검색은 `impact-analyzer`에게 맡기고, 당신은 스펙을 쓰고
결과를 검토·통합하고 판단합니다.

**유지보수 모드의 원칙**: 증상이 아니라 원인을 고칩니다. 최소 변경만 합니다 — 리팩토링이나 범위
밖 개선은 하지 않습니다. "고쳤다"는 주장은 항상 재현 테스트로 뒷받침합니다 — 재현 테스트 없이
고치면, 나중에 정말 고쳐졌는지 확인할 방법이 없습니다.

**`<run>` 표기 규칙**: 이 문서에서 `<run>`은 1단계에서 만든 실제 run 폴더 이름을 가리키는
자리표시자입니다 (`runs/README.md` 참고). 서브에이전트에게 보고 경로를 지시할 때는 이 값을
실제 이름으로 치환해서 넘깁니다 — 확실하지 않으면 `runs/.current`를 다시 읽습니다.

**서브에이전트 호출 규칙**: 아래에서 `code-generator`/`impact-analyzer`를 Agent 도구로 부를
때마다, `description`을 항상 `"[<task_id>] <짧은 설명>"` 형식으로 씁니다. `update_board.py`
훅이 이 접두사로 보고 누락을 자동 감지합니다 — 5절 참고.

작업 대상: $ARGUMENTS

## 0. 사전 확인

- `.claude/project.yaml`이 없거나 `project.maturity`가 비어 있으면 먼저 `/init`을 실행하라고
  안내하고 멈춥니다.
- `project.source_root`(기본값 `src`) 아래를 Glob으로 확인합니다. 폴더가 없거나 `README.md`
  정도만 있고 실제 코드가 사실상 없으면, 고칠 대상 자체가 없다는 뜻입니다. `/create`를 먼저
  쓰라고 안내하고 멈춥니다.
- 지금 git 브랜치가 `.claude/project.yaml`의 `branches.protected`에 있으면, 사람에게 기능 전용
  브랜치를 만들고 전환해달라고 요청하고 멈춥니다. (`branch_guard.py` 훅이 실제 쓰기 시점에도
  한 번 더 막습니다.)

## 1. run 폴더 생성

**새 run을 만들기 전에, 진행 중인 run이 이미 있는지 확인합니다.** `runs/.current`가 있고 그
run의 `summary.md`에 `done`이 아닌 status가 남아있으면, 새로 만들기 전에 사람에게 "기존에
진행 중인 것으로 보이는 run(`<이전 run>`)이 있습니다 — 이어서 할까요, 새로 시작할까요?"라고
물어봅니다. `runs/.current`는 전역 포인터 하나뿐이라, 같은 저장소를 동시에 여는 다른 세션이
있으면 나중에 만든 run이 먼저 것을 덮어씁니다 (`.claude/decisions/0011` 참고).

**타임스탬프는 직접 만들어내지 않습니다** — 반드시 `date` 명령으로 실제 값을 받아옵니다. Bash로
아래 명령을 실행합니다 (`SLUG`만 이번 작업에 맞는 짧은 이름 — 소문자·숫자·하이픈만 — 으로
바꿉니다):

```bash
SLUG="문제-슬러그"
RUN_ID="$(date +%Y%m%d-%H%M%S)_${SLUG}_$(printf '%04x' $((RANDOM % 65536)))"
mkdir -p "runs/$RUN_ID/agents"
printf '%s' "$RUN_ID" > runs/.current
printf 'maintain' > "runs/$RUN_ID/MODE"
echo "RUN_ID=$RUN_ID"
```

이 명령의 출력(`RUN_ID=...`)이 이후 이 문서에서 말하는 `<run>`의 실제 값입니다. (`MODE`가
`create`가 아니므로 `approval_gate.py`의 승인 게이트는 이 run에 적용되지 않습니다 —
`/maintain`은 원래 승인 절차 없이 진행합니다.) 이후 모든 서브에이전트 보고는 이 폴더 밑에
쌓이고, `update_board.py` 훅이 `summary.md`를 자동으로 갱신합니다.

## 2. 재현 — 실패하는 테스트로 버그를 먼저 고정합니다

버그를 고치기 전에, 그 버그가 실제로 존재한다는 걸 증명하는 테스트를 먼저 만듭니다.

- `task_id`를 하나 정합니다 — 이번 run에서 처음 쓰는 값 (예: `t1`).
- `code-generator`에게 위임할 때 **반드시 명시적으로 뒤집어서** 알립니다: "이 작업은 버그를
  재현하는 테스트를 작성하는 것이 목적입니다. 이 테스트는 지금 버그가 있는 코드에 대해
  **실패해야 정상**입니다. 훅이 Write 직후 테스트 실패를 보고하더라도 고치려 하지 말고,
  `status: done`으로 보고하고 summary에 '재현 테스트 작성 완료 — 의도대로 실패함'이라고
  적으세요." — `code-generator`의 기본 지침("훅이 실패를 알려주면 그에 맞춰 수정한 뒤 다시
  보고")은 이번 작업과 정반대이므로, 스펙에서 명시적으로 뒤집지 않으면 서브에이전트가 존재하지
  않는 버그를 고치려고 헤맬 수 있습니다.
- 보고를 받으면, 정말로 테스트가 실패했는지(훅이 남긴 메시지나 보고서의 `verification` 내용으로)
  직접 확인합니다. 실패하지 않았다면 재현이 안 된 것이므로, 테스트를 다시 작성하게 합니다.

## 3. 원인 분석 + 영향범위 지도 작성

`/extend`와 달리 유지보수에서는 영향범위 분석이 조건부가 아니라 **항상 필요**합니다 — 기존
동작을 건드리는 것 자체가 리스크이기 때문입니다.

- 새 `task_id`를 정합니다 — 2절과 겹치지 않게 (예: `t2`).
- `impact-analyzer`에게 위임합니다: 고치려는 코드가 어디서 참조되는지, 이 수정이 건드릴 수 있는
  다른 모듈이 무엇인지 찾게 합니다. 검색 범위는 `project.source_root` 아래, 정확한 보고 경로는
  `runs/<run>/agents/impact-analyzer_<task_id>.md`.
- 결과를 바탕으로 원인을 분석합니다 — 겉으로 보이는 증상이 아니라 근본 원인을 짚습니다.

## 4. 최소 변경 스펙 작성 → 가벼운 확인 → Generator 위임

- 새 `task_id`를 정합니다 — 이전 값들과 겹치지 않게 (예: `t3`).
- 스펙에 다음을 전부 포함합니다:
  - 파악한 근본 원인
  - 고쳐야 할 최소 범위 — **이것만 바꾸고 그 이상은 건드리지 말라**는 지시 포함
  - 2절에서 만든 재현 테스트를 통과시켜야 한다는 것
  - 3절의 영향범위 분석 결과
  - 정확한 파일 경로 (`project.source_root` 기준 — 오케스트레이터가 직접 정합니다)
  - 정확한 보고 경로 (`runs/<run>/agents/code-generator_<task_id>.md`)

**code-generator를 호출하기 전에, 근본 원인·최소 변경 범위·스펙을 사람에게 보여주고 진행
승인을 기다립니다.** `/extend` 4절과 동일한 이유로 훅으로 강제되지는 않습니다 — 대화창에서
명시적인 진행 승인을 받은 뒤에만 다음으로 넘어갑니다. 특히 유지보수는 "증상이 아니라 원인을
고친다"는 원칙이 걸려 있으므로, 사람이 원인 분석 자체에 동의하는지 확인하는 게 스펙 자체의
정확도보다 더 중요할 수 있습니다.

승인을 받으면 `code-generator`에게 위임합니다. 이번에는 `code-generator`의 기본 지침(훅이
실패를 알려주면 고쳐라)이 정상 작동합니다 — 목표는 재현 테스트와 기존 테스트가 전부 통과하는
상태입니다.

## 5. 결과 확인

`/extend` 5절과 동일합니다 — 훅이 자동으로 테스트·린트를 돌리고 `summary.md`를 갱신하며, 보고
누락도 자동으로 감지합니다. 여기서는 추가로 다음을 확인합니다:

- 2절에서 만든 재현 테스트가 이제 **통과**하는지
- 그 외 기존 테스트가 이번 수정으로 새로 깨지지 않았는지

`status: blocked`나 `failed`가 있으면 원인을 보고 재지시하거나, 하네스가 판단할 수 없는 문제면
사람에게 에스컬레이션합니다.

## 6. 판단이 필요한 검증만 verifier에게

`/extend` 6절과 동일합니다 (API 하위 호환성 판단, flaky 테스트 여부 판단, 문서화되지 않은 설계
의도와의 정합성 판단). 유지보수 모드에서는 특히 **수정과 무관해 보이는 다른 테스트가 실패했을
때** — 이번 변경 때문인지 원래 불안정한 테스트였는지 판단하는 경우가 자주 필요합니다.

## 7. 마무리

- `runs/<run>/summary.md`가 전부 `done`인지 확인합니다.
- `.claude/decisions/`에 짧게 기록합니다 — 원인이 무엇이었는지, 왜 이렇게 고쳤는지. 팀이 나중에
  비슷한 문제를 또 겪었을 때 참고할 수 있게 합니다.
- 변경 요약(원인 → 수정 → 재현 테스트 통과 확인)과 함께 사람에게 최종 확인을 요청합니다 — 훅은
  테스트/린트까지만 자동으로 걸러주고, 최종 승인은 사람의 몫입니다.
