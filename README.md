# harness_v1

오케스트레이션 기반 AI 코딩 하네스. Claude Code 세션(오케스트레이터) + 역할이 좁게 고정된
서브에이전트 + 훅(코드)으로 규칙을 강제하는 구조로, "AI가 알아서 잘 하겠지"에 기대지 않고
사람이 개입해야 하는 지점과 기계적으로 막아야 하는 규칙을 미리 설계해둔 템플릿입니다.

이 브랜치(`main`)는 특정 프로젝트가 아니라 **재사용 가능한 템플릿**입니다. 실제 프로젝트는
`/init`으로 매니페스트를 채운 뒤 별도 브랜치(예: `project/<프로젝트명>`)에서 진행합니다.

## 핵심 아이디어 — 누가 무엇을 하는가

| 주체 | 역할 |
|---|---|
| **사람** | 모드 선택과 최초 설정(`/init`에서 신규/레거시 선언), `/create`의 아키텍처 승인(파일로 직접), `/extend`·`/maintain`의 스펙에 대한 가벼운 진행 승인(대화창), 브랜치 생성/전환, 최종 검토 |
| **오케스트레이터(메인 세션)** | 코드를 직접 쓰지 않습니다. 컨벤션 파악, 영향범위 판단, 스펙 작성, 서브에이전트 결과 검토·통합, 사람에게 확인 요청 |
| **서브에이전트** | `code-generator`/`impact-analyzer`/`verifier` — 역할이 좁게 고정되어 있고, 스펙에 없는 판단(리팩토링, 범위 확장)은 하지 않습니다 |
| **훅(파이썬 스크립트)** | 사람도 오케스트레이터도 우회할 수 없는 규칙 — 테스트/린트, 보호 브랜치 쓰기 금지, `/create` 아키텍처 승인, 보고 누락 감지를 코드로 강제 |

가장 중요한 구분: **`/create`의 승인은 훅으로 강제되고, `/extend`·`/maintain`의 승인은 대화로만
확인됩니다.** 신규 프로젝트 전체 구조는 되돌리기 비용이 커서 하드 게이트를 걸었고, 기존 코드
위의 작은 변경은 이미 테스트·린트라는 안전망이 있어 가벼운 확인으로 충분하다고 판단했기
때문입니다 (`.claude/decisions/0007-extend-maintain-light-spec-confirmation.md` 참고).

## 시작하는 법

```
1. /init                          → .claude/project.yaml 채우기 (신규/레거시, 스택, 테스트·린트 명령)
2. 사람이 기능 전용 브랜치로 전환   → 하네스가 자동으로 만들지 않습니다
3-a. 신규 프로젝트 → /create "만들려는 것에 대한 설명"
3-b. 기존 코드에 기능 추가 → /extend "추가할 기능"
3-c. 버그 수정 → /maintain "고칠 문제"
```

`/init`을 가장 먼저 실행해야 합니다 — `project.maturity`(new/legacy)를 사람이 한 번 선언하면
이후 모든 모드가 그 값을 그대로 참조하고, 런타임에 추측하지 않습니다.

## 4가지 모드

| 명령 | 용도 | 사람 개입 지점 | 강제 방식 |
|---|---|---|---|
| `/init` | 매니페스트(`.claude/project.yaml`) 생성/갱신 | 성숙도·스택·테스트 명령 확인 | 없음 (대화형 도구) |
| `/create` | 신규 프로젝트 뼈대 생성 | 아키텍처 제안서(`PROPOSAL.md`) 승인 — `runs/<run>/APPROVED` 파일을 **사람이 직접** 생성 | **훅 강제** (`approval_gate.py`) |
| `/extend` | 기존 코드에 기능 추가 | 스펙을 보고 대화창에서 진행 승인 | 대화 기반 (훅 아님) |
| `/maintain` | 버그를 재현 테스트로 고정 후 최소 수정 | 원인 분석 + 최소 변경 스펙에 대한 진행 승인 | 대화 기반 (훅 아님) |

`/create`는 신규 프로젝트(`project.maturity: new`) 전용이고, `/extend`·`/maintain`은 이미 코드가
있어야 동작합니다 — 서로 반대 전제를 검사하므로 잘못 부르면 다른 모드를 쓰라고 안내하고
멈춥니다.

## 서브에이전트

| 이름 | 기본 모델 | 역할 | 도구 |
|---|---|---|---|
| `code-generator` | haiku | 오케스트레이터가 준 스펙만 정확히 구현. 모호하면 임의로 채우지 않고 `blocked`로 보고 | Read, Write, Edit, Grep, Glob |
| `impact-analyzer` | haiku | 심볼 참조를 코드베이스 전체에서 검색. 안전한지 판단하지 않고 목록만 만듦 | Read, Grep, Glob, Write |
| `verifier` | sonnet | 훅으로 기계적으로 판정할 수 없는 것만 — 하위 호환성 파괴 여부, flaky 테스트 판별, 문서화 안 된 설계 의도 정합성 | Read, Grep, Glob, Bash, Write |

모델 배정은 `.claude/project.yaml`의 `models.*`로 덮어쓸 수 있습니다 (비워두면 위 기본값).
셋 다 작업이 끝나면 `runs/<run>/agents/<이름>_<task_id>.md`에 정해진 형식으로 보고를 남기고
종료합니다 — 이 보고 없이는 오케스트레이터가 뭘 했는지 알 방법이 없습니다.

## 훅 — 기계적으로 강제되는 규칙 (`.claude/hooks/`)

| 훅 | 시점 | 하는 일 |
|---|---|---|
| `branch_guard.py` | Write/Edit 전 | 지금 브랜치가 `branches.protected`에 있으면 쓰기 자체를 차단 |
| `approval_gate.py` | Write/Edit 전 | `runs/<run>/APPROVED`는 하네스 도구로 절대 못 만듦(사람만 가능). `/create` run에서는 이 파일이 있어야 `source_root` 아래에 쓸 수 있음 |
| `post_write_check.py` | Write/Edit 후 | 매니페스트의 `commands.test`/`commands.lint`를 실행, 실패하면 알리고 다음 단계로 못 넘어가게 함 |
| `update_board.py` | 서브에이전트 종료 시 | `runs/<run>/summary.md`를 재생성. Agent 호출의 `description`이 `"[<task_id>] ..."` 형식이면, 보고 누락도 자동 감지해 경고로 표시 |

이 규칙들은 프롬프트 지시가 아니라 코드이므로, 오케스트레이터가 서두르거나 착각해도 뚫리지
않습니다. 반대로 `/extend`·`/maintain`의 스펙 확인처럼 "가벼운 확인"으로 정한 지점은 의도적으로
훅에 넣지 않았습니다 — 작은 변경까지 매번 파일 기반 게이트를 걸면 두 모드의 존재 이유(승인
게이트 없이 빠르게 간다)가 사라지기 때문입니다.

## 진행 상황 확인 — `runs/`

```
runs/<YYYYMMDD-HHMMSS>_<슬러그>_<해시>/
  MODE               # create | extend | maintain — 훅이 어떤 규칙을 적용할지 여기로 판단
  PROPOSAL.md        # /create만 — 아키텍처 제안서
  APPROVED           # /create만 — 사람이 직접 만들어야 생기는 승인 마커
  summary.md         # 자동 생성 — task_id별 최신 status. 오케스트레이터가 매턴 읽음
  agents/
    <서브에이전트명>_<task_id>.md   # 상세 보고 (디버깅·감사용)

runs/.current        # 지금 활성 run 폴더 이름
```

`summary.md`는 직접 편집하지 않습니다 — `update_board.py`가 서브에이전트 종료마다 다시 씁니다.
이 폴더는 버전 관리에 포함해 감사 기록으로 남기는 걸 권장합니다 (`runs/README.md` 참고).

## 매니페스트 — `.claude/project.yaml`

모든 세션이 참조하는 단일 진실 소스입니다. `/init`이 스캔 + 질문으로 채우고, 사람이 직접 고쳐도
됩니다.

- `project.maturity` — `new`/`legacy`. 하네스는 이 값을 런타임에 추측하지 않습니다.
- `commands.test`/`commands.lint` — `post_write_check.py`가 그대로 실행합니다. 비워두면 해당
  게이트는 조용히 건너뜁니다.
- `branches.protected` — `branch_guard.py`가 참조. 브랜치를 자동으로 만들지는 않습니다.
- `models.*` — 서브에이전트 모델 오버라이드. 채우면 `/init`이 해당 `.claude/agents/*.md`의
  `model:` 프론트매터를 덮어씁니다.

## 결정 로그 — `.claude/decisions/`

모드/아키텍처와 관련된 설계 결정을 기록하는 곳입니다. "왜 이렇게 만들었는지"가 코드만 봐서는
안 보일 때(트레이드오프, 기각한 대안, 발견된 한계) 여기 남깁니다. 새 프로젝트를 시작하면
프로젝트 특유의 결정은 그 프로젝트 브랜치에, 하네스 자체의 설계 결정은 여기 계속 쌓입니다.

## 자주 헷갈리는 점

- **브랜치는 하네스가 자동으로 만들지 않습니다.** 이름도 팀이 정합니다 — `branch_guard.py`는
  "지금 보호된 브랜치에 있다"는 사실만 차단할 뿐, 대안 브랜치를 대신 만들어주지 않습니다.
- **`/create`만 하드 게이트가 있습니다.** `/extend`·`/maintain`은 대화창 확인만으로 진행되므로,
  사람이 실제로 답하기 전까지 다음 단계로 넘어가지 않는지는 오케스트레이터의 판단에 달려
  있습니다 (프롬프트 인젝션 등 극단적 상황에 대한 방어력은 `/create`보다 낮습니다 — 의도된
  트레이드오프입니다).
- **제품 코드는 `project.source_root`(기본 `src`) 아래에만 둡니다.** `.claude/`, `runs/`는
  하네스 자체이므로 컨벤션 파악·영향범위 검색 대상에서 제외됩니다.
